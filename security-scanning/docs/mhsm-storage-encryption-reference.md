## 1. Envelope Encryption Model

> Source: [Doc #1], [Doc #2], [Doc #8]

Azure Storage uses **envelope encryption** — a two-tier key model where data is encrypted
with a Data Encryption Key (DEK), and the DEK itself is encrypted (wrapped) with a Key
Encryption Key (KEK) stored in a Managed HSM.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     ENVELOPE ENCRYPTION MODEL                                │
│                     (Source: Doc #1, Doc #8)                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                               │
│   ┌─────────────────────────────┐                                            │
│   │  Blob / File / Table / Queue │  ← Your data (plaintext)                  │
│   └──────────────┬──────────────┘                                            │
│                  │                                                            │
│                  │  Encrypted with                                            │
│                  ▼                                                            │
│   ┌─────────────────────────────┐                                            │
│   │  DEK (Data Encryption Key)  │  ← AES-256 symmetric key                  │
│   │  Generated per-account or   │  ← Unique per storage account              │
│   │  per-blob for large objects │  ← Never stored in plaintext               │
│   └──────────────┬──────────────┘                                            │
│                  │                                                            │
│                  │  Wrapped (encrypted) with                                  │
│                  ▼                                                            │
│   ┌─────────────────────────────┐                                            │
│   │  KEK (Key Encryption Key)   │  ← RSA-HSM 3072/4096 in Managed HSM       │
│   │  Customer-Managed Key (CMK) │  ← Customer controls lifecycle             │
│   │  Stored in Managed HSM      │  ← FIPS 140-3 Level 3 boundary            │
│   └─────────────────────────────┘                                            │
│                                                                               │
│   Storage at rest = { EncryptedData, EncryptedDEK, KeyVaultURI, KeyVersion } │
│                                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

**Key properties** (from [Doc #2]):
- The DEK is generated by the Azure Storage service
- The KEK never leaves the HSM boundary — only wrap/unwrap operations are performed
- The storage account stores the wrapped (encrypted) DEK alongside the encrypted data
- Without access to the KEK, the DEK cannot be recovered and data is inaccessible

## 5. Key Rotation Flow

> Source: [Doc #5], [Doc #6]
>
> **Note:** As of current documentation, Managed HSM supports rotation policies
> with notifications. Azure Storage automatically detects the new key version
> when `keyVersion` is omitted from the storage account configuration (Doc #3).

```

│                      KEY ROTATION FLOW                                        │
│                      (Source: Doc #5, Doc #6, Doc #3)                         │

│             │                                                                 │
│             │  Generates new key version                                      │
│             ▼                                                                 │
│  ┌─────────────────────────────────────────────────────────┐                 │

│  │                                                          │                 │
│  │  Key: storage-encryption-key                             │                 │
│  │  • Version N (old) — still active for existing wraps    │                 │
│  │  • Version N+1 (new) — used for new wrap operations     │                 │
│  └──────────┬──────────────────────────────────────────────┘                 │
│             │                                                                 │
│             │  Event Grid event: "KeyNewVersionCreated" (Doc #6)             │
│             ▼                                                                 │

│  │                                                          │                 │

│  └──────────┬──────────────────────────────────────────────┘                 │
│             │                                                                 │

│  │  Azure Storage — Background Re-Wrap (Doc #3)            │                 │
│  │                                                          │                 │
│  │  For each stored EncryptedDEK:                          │                 │
│  │                                                          │                 │
│  │    ┌───────────────────────────────────────────┐        │                 │
│  │    │ 1. Unwrap EncryptedDEK with key version N │        │                 │
│  │    │    (old version) → get plaintext DEK      │        │                 │
│  │    │                                           │        │                 │
│  │    │ 2. Wrap DEK with key version N+1          │        │                 │
│  │    │    (new version) → new EncryptedDEK       │        │                 │
│  │    │                                           │        │                 │
│  │    │ 3. Update blob metadata with new          │        │                 │
│  │    │    EncryptedDEK + new version reference   │        │                 │
│  │    └───────────────────────────────────────────┘        │                 │
│  │                                                          │                 │
│  │  Properties:                                             │                 │
│  │  • Transparent — no client downtime                     │                 │
│  │  • Actual data NOT re-encrypted (same DEK)             │                 │
│  │  • Old key version retained until all re-wraps complete │                 │
│  │  • Duration: hours for large accounts (PBs of data)     │                 │
│  └─────────────────────────────────────────────────────────┘                 │
│                                                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 9. Failure & Revocation Scenarios

> Source: [Doc #2], [Doc #4], [Doc #11], [Doc #13]

| Scenario | Impact | Recovery | Source |
|----------|--------|----------|--------|
| **mHSM region outage** | Storage I/O fails (wrap/unwrap unavailable). Data intact but inaccessible. | Wait for region recovery. Deploy multi-region HSM for HA. | Doc #4, #13 |
| **KEK disabled** | All wrap/unwrap denied. Storage returns 403 on read/write. | Re-enable key in mHSM (immediate). | Doc #2 |
| **KEK soft-deleted** | Access denied. Key in soft-delete state. | Recover key within retention period (7-90 days). | Doc #4, #11 |
| **KEK purged** (no purge protection) | **PERMANENT DATA LOSS.** Crypto-shred — no recovery path. | None. Enable purge protection to prevent. | Doc #11, #13 |
| **Managed Identity deleted** | Storage cannot authenticate to mHSM. I/O fails. | Recreate MI, reassign RBAC role on HSM. | Doc #3 |
| **RBAC role revoked** | Immediate wrap/unwrap denial. Storage I/O fails. | Reassign Crypto Service Encryption User role. | Doc #12 |
| **Normal key rotation** | Transparent. Old version serves existing DEKs. Background re-wrap uses new version. | No action needed. Automatic. | Doc #5, #3 |
| **HSM security domain lost** | Cannot recover HSM in disaster scenario. | Maintain offline backup of security domain (3-of-5 quorum). | Doc #4, #13 |
